# 開発運用ルール (RULES.md)

## 1. 基本原則

1.  **JSON First**: すべてのコンテンツとUIテキストはJSONファイルで管理する。ハードコード禁止。
2.  **型安全性**: TypeScriptの型チェックを厳格に適用する。`any` は原則禁止。
3.  **没入感**: ユーザー体験を最優先し、リッチな表現とパフォーマンスを両立させる。

## 2. 実装ルール

### 2.1 コンポーネント
*   **配置**: `app/components/` 配下の適切なディレクトリ（`core`, `modules`, `templates`）に配置する。
*   **命名**: パスカルケース（例: `FullscreenHero.tsx`）。機能を表す具体的な名前をつける。
*   **依存**: `core` コンポーネントは他のコンポーネントに依存してはならない。

### 2.2 スタイリング
*   **Tailwind CSS**: 原則としてTailwindのユーティリティクラスを使用する。
*   **レスポンシブ**: モバイルファーストで記述する（例: `w-full md:w-1/2`）。
*   **テーマ変数**: 色やフォントサイズはハードコードせず、Tailwindの設定（`bg-primary-500`など）を使用する。

### 2.3 画像
*   **next/image**: 必ず `next/image` コンポーネントを使用する。
*   **Alt属性**: 意味のある代替テキストを必ず設定する。装飾用画像は空文字 `alt=""` とする。
*   **パス**: 画像パスの直接指定は避け、`lib/images.ts` 経由で解決する。

## 3. Git / ワークフロー運用

### 3.1 コミット
*   **粒度**: 機能単位、タスク単位で細かくコミットする。
*   **メッセージ**: プレフィックスをつける（`feat:`, `fix:`, `docs:`, `style:`, `refactor:`）。
    *   例: `feat: add fullscreen hero component`
*   **秘匿**: ベンチマークサイトの具体的な名称をコミットメッセージやPRに含めない。「Reference Design」等の表現を使用する。

### 3.2 プッシュ
*   **禁止事項**: `main` ブランチへの直接プッシュは禁止。
*   **手順**: ユーザー（管理者）の指示を受けてからプッシュする。

### 3.3 タスク管理
*   **TODO**: 作業開始時にTODOリストを作成・更新する。
*   **ログ**: 作業完了時に `WORK_LOG.md` を更新する。

## 4. 品質管理

### 4.1 チェックリスト
プッシュ前に必ず以下を確認すること。

- [ ] `npm run lint` がエラーなしで通過するか
- [ ] `npm run build` がエラーなしで通過するか
- [ ] コンソールにエラーや警告が出ていないか
- [ ] レスポンシブ表示（PC/スマホ）が崩れていないか

### 4.2 禁止事項
*   ビルドエラーを放置したままのコミット。
*   未使用のコードやコメントアウトされたコードの放置。
*   `console.log` の残存（デバッグ用）。
